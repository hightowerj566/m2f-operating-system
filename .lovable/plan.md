
# M2F Flagship Guided Journey — Day-Based Engine

Rebuild the flagship program around a continuous **252-day pre-birth calendar** (Program Day 1 = 252 days before due date) plus an open-ended post-birth day count, resolved from the member's due date / birth date / today.

## 1. Data model

New file: `src/data/m2f_flagship_journey.json` — schemaVersion `2.0.0`, `preBirthJourneyLengthDays: 252`.

Contains:
- `stages[]` — 5 pre-birth + 3 post-birth, each with `programDayStart/End` (contiguous, no gaps).
  - Foundation 1–70, Framing 71–133, Durability 134–196, Staging 197–224, Mission Mode 225–252.
  - Post-birth: Survival 1–35, New Dad Foundation 36–84, Father Athlete 85+.
- `days[]` — one entry for every integer 1…252. Fields: `programDay`, `relativeDaysToDueDate`, `stageId`, `stageDay`, `weekNumber`, `dayOfWeekInCycle`, `dayType`, `title`, `objective`, `isRequired`, `workoutId?`, `activities?`, `completionCriteria?`, `completionMessage?`, `estimatedDurationMinutes`.
- `workouts[]` — training-day sessions, each with `full/express/minimum` versions referencing `exerciseId`s from the existing library.
- `postBirthDays[]` — a 28-day rolling template per post-birth stage that we tile across `postpartumDay` (survival gets its own bespoke first 2 weeks).

Days are generated by a small build script (co-located, run at import time in Node/tsx) — not hand-written 252 times — but the resulting JSON is committed so runtime is pure data lookup. Weekly cycle templates:

- Foundation (weeks 1–10, days 1–70): Upper A, Lower A, Active Recovery, Upper B, Lower B, Pump & Athletic, Full Rest.
- Framing (weeks 11–19, days 71–133): Upper Strength, Lower Strength, Active Recovery, Upper Strength+Grip, Lower Strength+Carries, Upper Pump+Athletic, Full Rest.
- Durability (weeks 20–28, days 134–196): Upper Strength, Lower Strength, Active Recovery+Mobility, Upper Durability, Lower Durability, Optional Zone 2+Core, Full Rest.
- Staging (weeks 29–32, days 197–224): Full Body A, Recovery, Full Body B, Mobility, Full Body C, Short Pump+Carries, Full Rest.
- Mission Mode (weeks 33–36, days 225–252): Full Body A, Rest/Walk, Full Body B, Mobility/Rest, Full Body C, Optional Walk, Full Rest.

Existing JSON workouts from `m2f_training_programs.json` are mapped in where they cover the same slot; new rest / recovery / optional / mobility day objects are synthesized with `dayType`, `activities`, and `completionCriteria` per spec.

## 2. Resolver

New file: `src/lib/training/getFlagshipJourneyDay.ts`

```ts
type FlagshipJourneyDayResult =
  | { status: "active"; programDay; daysUntilDueDate; stageId; stageDay; programWeek; dayContentId; scheduledDate }
  | { status: "pre-program"; programDay: 1; daysUntilProgramStart; scheduledStartDate }
  | { status: "post-due-date"; daysPastDueDate; awaitingBirthConfirmation: true }
  | { status: "post-birth"; postpartumDay; stageId; dayContentId }
  | { status: "needs-due-date" };
```

Uses `date-fns` `differenceInCalendarDays` + `startOfDay` for timezone-safe calendar math. Formula:

```ts
const PRE_BIRTH_JOURNEY_LENGTH_DAYS = 252;
const days = differenceInCalendarDays(startOfDay(dueDate), startOfDay(today));
const raw = PRE_BIRTH_JOURNEY_LENGTH_DAYS - days + 1;
const programDay = Math.min(252, Math.max(1, raw));
```

Selection priority (applied here + wired in `useMemberProgram`):
1. Active coach assignment (non-flagship) → coach lane, but resolver still runs in background so `flagshipProgramDay` is preserved.
2. `babyArrivedAt` set → post-birth branch (`postpartumDay = diff(today, birth) + 1`).
3. Due date passed, no birth → `birth-window` (`post-due-date`).
4. Due date within 252 days → `active`.
5. Due date >252 days out → `pre-program`.
6. No due date → `needs-due-date`.

- Assert every integer 1…252 present exactly once, contiguous.
- Assert every `stageId` maps to a stage and `stageDay` is in range.

- Assert stage ranges are contiguous with no overlaps.
- Existing exerciseId library check preserved.
